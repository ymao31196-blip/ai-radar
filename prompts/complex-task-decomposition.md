# 复杂任务分解 — 从 GPT-5.5 到通用实践

**日期**：2026-05~06
**来源**：OpenAI GPT-5.5 官方指南 + OpenAI Cookbook
**可信度**：🟢 官方推荐 + 🟡 社区验证

### 适用任务
- 任务描述：复杂编程任务、多步骤分析、需要多种工具协作的场景
- 适用模型：GPT-5.5（原生优化）/ Claude 4 / DeepSeek V4 / 通用推理模型

### Prompt 模板

#### 模板 A：通用复杂任务分解（适用于所有推理模型）

```
请按以下步骤完成此编程任务：

(1) 理解需求 — 列出需要实现的功能点，确认是否有歧义
(2) 设计接口 — 设计数据结构和函数签名
(3) 实现核心逻辑 — 逐步推理，每步验证
(4) 边界处理 — 添加错误处理和边界情况

在每一步开始前，先发送一条简短的状态更新（"正在进行第X步：..."）。

任务描述：[输入任务]
```

#### 模板 B：GPT-5.5 专用最小 Prompt（从零开始原则）

```
[仅包含任务描述，不添加任何额外指令]

[输入任务]
```

> GPT-5.5 官方指南强调：从「能保持产品契约的最小 Prompt」开始，不要从旧模型 Prompt 直接迁移。让模型自己推理比过度引导效果更好。

#### 模板 C：Codex CLI `/goal` 模式

```bash
# 在 Codex CLI 中
/goal: 完成 [功能描述] 并确保所有测试通过
```

系统将循环执行直到目标完成或 token 预算耗尽。

### 关键技巧
- **最小 Prompt 原则**（GPT-5.5）：从最小指令开始，逐步添加约束。过度引导会抑制模型自身推理能力
- **中间状态可见**：多步骤任务中，要求在调工具前发送简短状态更新（「正在分析您的请求，第一步是...」）
- **从零重写 Prompt**：不要从旧模型（GPT-4o、GPT-4）直接迁移 Prompt。GPT-5.5 对 Prompt 的敏感度与之前完全不同
- **负面约束仍有效**：Codex 的 base_instructions 包含大量负面约束（「never talk about goblins...」），说明在特定场景下明确的禁止规则仍然重要

### 为什么有效
OpenAI Cookbook 核心原则"Split complex tasks into simpler subtasks"——每个原子步骤比完整任务更可靠。GPT-5.5 优化了自身推理链的自主规划能力，过度引导反而破坏这个能力。

### 变体 & 调优
- **非推理模型**（GPT-4o、Claude 3.5 等）：保留明确的步骤分解引导（模板 A），不要用最小 Prompt
- **状态更新频率**：对低延迟模型（DeepSeek V4 Flash）每步都更新；对高延迟模型（Opus 4.8）仅在关键步骤间更新
- **Token 预算**：复杂任务设总 token 上限，单步骤超限时自动降级到更简单的子问题

| 模型 | 效果 | 备注 |
