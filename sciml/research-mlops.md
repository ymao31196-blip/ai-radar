# 科研 MLOps 工具

> 覆盖：Weights & Biases、MLflow、DVC
> 研究生阶段的实验管理：追踪每次跑的结果、管理数据集版本、避免「这个实验结果存在哪个文件夹」的困境。

---

## 为什么需要科研 MLOps

科研实验的特点：
- 每天跑几十组实验，超参数组合爆炸
- 实验结果碎片化：这个结果在 notebook 里，那个在终端输出里
- 数据预处理步骤不可复现：上次用的清洗脚本是哪个版本？
- 论文写完了才发现某个对比实验的种子没记下来

三个工具可以覆盖以上问题，而且不互斥，可以叠加使用。

## Weights & Biases (wandb)

**类型**：实验追踪 + 可视化
**官网**：https://wandb.ai
**可信度**：🟢 正式产品

### 是什么
最流行的 ML 实验追踪工具。自动记录超参数、loss 曲线、模型权重、硬件利用率、预测样本。web 仪表盘可直接对比不同实验的曲线。

### 对科研有什么用
- **一键记录**：`wandb.init()` + `wandb.log()` 三行代码，自动记录所有实验参数和指标
- **画图不用自己写**：loss 曲线、学习率变化、预测 vs 真实值对比图自动生成
- **团队协作**：课题组可以共享一个 project，互相看实验结果，避免重复跑
- **论文就绪**：实验表格直接导出，指标对比图可直接截图放入论文

### 关键信息

| 维度 | 内容 |
|------|------|
| 核心功能 | 实验追踪、超参数记录、权重存储、可视化 |
| 集成难度 | 极低（pip install + 3 行代码） |
| 免费额度 | ✅ 个人/学术免费，无限项目 |
| 数据存储 | 云端（也可自托管） |
| 代码 | https://github.com/wandb/wandb |

### 适用场景
- 超参数调优阶段（不同 learning rate / 模型大小 / 数据量的对比）
- 论文实验数据存档（保证评审时可复现）

### 局限性
- 数据存云端，敏感数据需注意
- 多 GPU 训练需额外配置
- 如果不连网，日志记录会阻塞

- [ ] 把现有实验接入 wandb 追踪
- [ ] 建立规范的项目命名体系（如 `sciml-wind/patchtst-3year`）

## MLflow

**类型**：ML 生命周期管理平台
**官网**：https://mlflow.org
**可信度**：🟢 正式产品

### 是什么
Linux Foundation 旗下的 ML 生命周期管理平台，涵盖实验追踪、模型注册、模型部署、模型 Registry。相比 wandb 更「重」，但覆盖面更广。

### 对科研有什么用
- **本地优先**：所有数据存在本地文件或自建服务器，适合科研数据不离开实验室的需求
- **模型注册**：每个实验的模型可以版本化注册，后续对比、复现、回溯都有明确记录
- **与 wandb 不冲突**：大多数团队同时用 wandb（实验追踪）+ MLflow（模型管理）

### 关键信息

| 维度 | 内容 |
|------|------|
| 核心功能 | 实验追踪 + 模型 Registry + 部署 |
| 集成难度 | 中（需配置 tracking server） |
| 免费额度 | ✅ 完全开源免费 |
| 数据存储 | 本地 / 自建服务器 / 云存储 |
| 代码 | https://github.com/mlflow/mlflow |

### 适用场景
- 课题组有自建 GPU 服务器，所有数据需要本地存储
- 需要模型版本化管理（不同实验的权重文件有组织地归档）

### 局限性
- 开箱体验不如 wandb 流畅，UI 相对朴素
- 多人协作需自建 tracking server
- 实验对比的交互式可视化不如 wandb

### 行动项
- [ ] 对比 wandb vs MLflow，选择主力实验追踪工具
- [ ] 推荐组合：wandb（日常实验追踪）+ MLflow（最终模型版本注册）
- [ ] 在课题组服务器上部署 MLflow tracking server

## DVC (Data Version Control)

**类型**：数据 + 模型版本管理
**官网**：https://dvc.org
**可信度**：🟢 正式产品

### 是什么
数据版本的 Git——用 Git 记录数据/模型的元数据（哈希指针），实际数据存储在云端或网络存储。`dvc pull` 即可恢复完整数据集。

### 对科研有什么用
- **数据集版本化**：「训练集 V1」和「训练集 V2（加入了 2026 年新数据）」在 DVC 中有明确记录，随时回退
- **pipeline 可复现**：谁用了什么预处理脚本、在哪个数据集上跑的、参数是什么——`dvc reproduce` 一键复现
- **与 Git 同源思维**：如果团队已经用 Git 管理代码，再加 DVC 管理数据几乎没有学习曲线
- **论文复现性**：提交论文时附带 `dvc.lock` 文件，评审人可直接复现

### 关键信息

| 维度 | 内容 |
|------|------|
| 核心功能 | 数据版本管理 + ML pipeline |
| 集成难度 | 中（需配置远程存储） |
| 免费额度 | ✅ 完全开源免费 |
| 数据存储 | S3/GCS/MinIO/网络磁盘/本地 |
| 代码 | https://github.com/iterative/dvc |

### 适用场景
- 数据集频繁更新（如持续接入新 SCADA 数据）
- 数据预处理链路复杂（SCADA 清洗 → 特征工程 → 归一化 → 数据集切分）
- 论文提交需要附带可复现的数据 + 代码版本

### 局限性
- 大文件（>1GB）的 `dvc push/pull` 速度取决于远程存储
- 多人同时修改 pipeline 时需注意冲突
- 小团队可能用不到全部 pipeline 功能

### 行动项
- [ ] 梳理当前项目的数据预处理流程，定义 DVC pipeline 阶段
- [ ] 配置远程存储（实验室 NAS 或 S3/MinIO）
- [ ] 提交论文时确保 `dvc.lock` + `Git commit` 可以完整复现

## 工具对比总结

| 维度 | Weights & Biases | MLflow | DVC |
|------|-----------------|--------|-----|
| 核心价值 | 实验追踪和对比 | 全生命周期管理 | 数据和 pipeline 版本化 |
| 与 wandb 的关系 | — | 互补（模型注册） | 互补（数据版本） |
| 免费 | ✅ 个人学术免费 | ✅ 完全开源 | ✅ 完全开源 |
| 学习成本 | **最低**（3行代码） | 中等 | 中-高 |
| 数据存储要求 | 云端 | 自建 | 自建 |
| 论文复现性贡献 | 实验记录 | 模型版本 | 数据 + pipeline 版本 |
| 80% 场景推荐 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

### 推荐组合

**最小配置**：wandb（实验追踪）
**完整配置**：wandb（实验追踪）+ DVC（数据/pipeline 管理）
**团队配置**：wandb + DVC + MLflow（模型 Registry）
